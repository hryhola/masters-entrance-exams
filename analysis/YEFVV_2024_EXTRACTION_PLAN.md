# План конвертації тесту ЄФВВ 2024 у YAML

## 1. Що містить PDF

**Джерело:** `YEFVV_2024-Informatsijni-tehnologiyi-na_sajt__.pdf`

- 29 сторінок формату A4;
- 140 пронумерованих завдань;
- кожне завдання має чотири варіанти відповіді;
- правильна лише одна відповідь;
- правильний варіант позначено маркером `(x)`, інші — `()`;
- відповіді не винесені в окремий ключ, а розміщені безпосередньо біля
  варіантів;
- документ має текстовий шар, тому OCR для основного тексту не потрібен;
- PDF не тегований;
- усередині є 45 растрових об'єктів, які використовуються для формул,
  таблиць, коду та схем.

Формальний тип усіх завдань один: **single choice**. Водночас спосіб
подання умови й варіантів суттєво відрізняється.

## 2. Виявлені види контенту

### Звичайний текст

Більшість завдань складаються з текстової умови та чотирьох текстових
варіантів. Наприклад, завдання 1, 4, 17, 28.

### Текст із фрагментами спеціальної нотації

- SQL-запити: 79, 83, 84, 101, 110;
- команди роботи зі стеком і чергою: 91, 126;
- двійкові числа й системи числення: 15, 40, 67, 131;
- математичні та системні позначення: 24, 61, 74, 120.

Такі фрагменти потрібно подавати як `code` або `math`, а не як звичайний
абзац.

### Візуально залежні завдання

Щонайменше 17 завдань втрачають суттєву частину змісту під час звичайного
витягу тексту:

| Завдання | Елемент |
|---:|---|
| 9 | чотири ER-діаграми як варіанти відповіді |
| 13 | математична границя |
| 16 | формули функцій активації у варіантах |
| 31 | набір із чотирьох формул |
| 38 | математична границя |
| 51 | клас і чотири фрагменти коду |
| 57 | функція та формули похідної |
| 71 | UML-діаграма |
| 75 | орієнтований граф |
| 76 | функція двох змінних і координати |
| 86 | таблиця реляційної бази даних |
| 97 | функція та формули похідної |
| 103 | вектори й матричний добуток |
| 113 | схема мережевої топології |
| 119 | орієнтований граф |
| 126 | фрагмент коду роботи з чергою |

Ці завдання потрібно перевіряти візуально за рендером сторінки. Покладатися
лише на `pdftotext` для них не можна.

## 3. Особливості, які впливають на конвертацію

### Завдання переносяться між сторінками

Наприклад:

- завдання 5 починається на сторінці 1, а останній варіант розміщено на
  сторінці 2;
- завдання 16 починається на сторінці 3 і продовжується на сторінці 4;
- завдання 26 починається на сторінці 5 і продовжується на сторінці 6;
- завдання 57 починається на сторінці 12 і завершується на сторінці 13;
- завдання 73, 77, 81, 101, 106, 116 і 126 також переходять на наступну
  сторінку.

Отже, розділення тексту за сторінками не може бути основним способом
сегментації.

### Службовий маркер повторюється на кожній сторінці

Текст `(x) – правильний варіант відповіді на завдання` у футері не є
частиною жодного завдання. Його потрібно видаляти до визначення правильної
відповіді.

### Формули витягуються неповністю

Частина математичних виразів зникає з текстового шару або розпадається на
фрагменти. Для фінального набору формули доцільно відтворити в LaTeX і
звірити з рендером PDF.

### Розбіжність між тестом 2024 і програмою 2025

Тест 2024 року містить конкретні мови програмування, операційні системи та
фрагменти програм. Програма, затверджена 2 грудня 2025 року, прямо
забороняє використання конкретних мов програмування, псевдокоду й
фрагментів програм у тестових завданнях.

Тому потрібно:

1. зберегти всі 140 офіційних завдань без вилучення;
2. окремо зіставити їх із програмою 2025 року;
3. позначити завдання, які не відповідають новим обмеженням або не мають
   прямої теми в новій програмі;
4. не використовувати такі завдання як шаблони для генерації тестів 2026
   без адаптації.

## 4. Рекомендована модель даних

YAML має бути єдиним файлом даних, але растрові ресурси слід зберігати
окремими файлами. Вбудовувати картинки в YAML як Base64 не варто.

Запропонована структура:

```text
dataset
  id
  title
  exam
  subject
  year
  language
  source
  question_count

questions[]
  id
  number
  type
  prompt[]
  options[]
    id
    content[]
  answer
    correct_option
    source
  explanation
  classification
    program_2025
    tags
  source
    file
    page_start
    page_end
    question_number
  review
```

### Блоки контенту

І `prompt`, і кожен `option.content` мають бути масивами блоків. Потрібні
такі типи:

| Тип | Призначення |
|---|---|
| `markdown` | абзаци, списки, звичайний текст |
| `math` | LaTeX для формул |
| `code` | SQL, код або команди зі збереженням відступів |
| `table` | заголовки та рядки як структуровані дані |
| `image` | шлях до схеми чи іншого графічного ресурсу та alt-текст |

Ця модель підтримує як просте текстове завдання, так і варіанти відповіді,
які повністю складаються із зображень або формул.

### Зображення

Рекомендована структура каталогів:

```text
data/yefvv-it-2024.yaml
assets/yefvv-it-2024/q009/option-a.png
assets/yefvv-it-2024/q009/option-b.png
assets/yefvv-it-2024/q071/prompt.png
assets/yefvv-it-2024/q140/prompt.png
```

Для кожного зображення потрібні:

- стабільний відносний шлях;
- роль: `prompt`, `option` або `reference`;
- короткий alt-текст;
- позначка про необхідність ручної перевірки візуального опису.

Прості таблиці треба переводити у структурований `table`, формули — у
`math`, а код — у `code`. Зображення слід залишати там, де графічна форма
має значення: ER/UML-схеми, графи, топології.

### Правильна відповідь

Правильна відповідь має посилатися на стабільний ідентифікатор варіанта:

```yaml
answer:
  correct_option: b
  source: official_marker
```

Маркер `(x)` не повинен залишатися в тексті варіанта.

### Пояснення

PDF не містить пояснень, тому на етапі транскрипції потрібно явно записати:

```yaml
explanation:
  status: missing
  blocks: []
```

Пояснення слід генерувати й перевіряти лише після завершення точної
транскрипції. Не можна змішувати офіційну відповідь із нашим поясненням.

### Редакторські виправлення

У PDF є друкарські помилки та неоднозначні формулювання. Їх не слід
виправляти мовчки. Допустимі автоматичні зміни:

- видалення переносів слів, спричинених версткою;
- об'єднання рядків у абзаци;
- нормалізація пробілів.

Зміни слів, формул або пунктуації потрібно фіксувати в `review.notes` або
окремому списку редакторських виправлень.

## 5. Чому не варто робити все одним промптом

Один промпт для 140 завдань створить одразу кілька ризиків:

- пропущені або здубльовані завдання;
- змішування варіантів сусідніх завдань на межах сторінок;
- неправильне визначення `(x)` через службовий футер;
- втрата формул і зображень;
- непомітне виправлення офіційного тексту;
- складна перевірка великого результату;
- помилки зіставлення з програмою 2025 року.

Краще розділити механічне витягування, змістову нормалізацію та
класифікацію.

### Етап 4. Нормалізація пакетами

Текстові завдання доцільно обробляти сімома пакетами по 20:

1. 1–20;
2. 21–40;
3. 41–60;
4. 61–80;
5. 81–100;
6. 101–120;
7. 121–140.

У межах кожного пакета:

- порівняти чернетку з PDF;
- відновити абзаци, математичну нотацію та форматування;
- перевірити рівно чотири варіанти;
- перевірити правильний варіант;
- записати сторінки джерела;
- не додавати пояснення чи виправлення змісту.

Візуально залежні завдання потрібно перевіряти окремою чергою по 1–5
завдань, а не разом із великим текстовим пакетом.

**Статус: завершено 9 червня 2026 року.**

- повторно запускний нормалізатор:
  `scripts/normalize-yefvv-2024.mjs`;
- нормалізований набір даних:
  `data/normalized/yefvv-it-2024.yaml`;
- еквівалентний JSON:
  `data/normalized/yefvv-it-2024.json`;
- звіт за пакетами та редакторськими примітками:
  `reports/yefvv-it-2024-normalization.json`.

Нормалізатор прибрав 214 версткових переносів рядка у 153 Markdown-блоках,
з'єднав 5 перенесених складених слів зі збереженням дефіса та залишив
багаторядковим лише нумерований список у завданні 107. SQL, макровизначення
і послідовності команд у завданнях 59, 79, 83, 84, 91 та 101 перетворено
на типізовані `code`-блоки.

Усі сім пакетів містять по 20 завдань і 80 варіантів. Окремо перевірено
15 завдань, що переносяться між сторінками. Офіційні відповіді, посилання
на сторінки, пояснення та класифікацію не змінено. Для 6 місць, де
оригінал містить незвичне написання або потребує предметної перевірки,
додано редакторські примітки без виправлення джерела.

YAML і JSON структурно еквівалентні. Повторний запуск створює побайтово
ідентичні нормалізовані файли та звіт.

### Етап 5. Автоматична валідація

Перевірки мають гарантувати:

- YAML синтаксично коректний;
- є рівно 140 завдань;
- номери утворюють послідовність `1..140`;
- усі `id` унікальні;
- кожне завдання має тип `single_choice`;
- кожне завдання має чотири унікальні варіанти;
- `correct_option` існує серед варіантів;
- жоден службовий `(x)` не потрапив у контент;
- усі сторінки джерела в межах `1..29`;
- усі посилання на зображення існують;
- усі візуально залежні завдання мають ручну перевірку.

**Статус: завершено 9 червня 2026 року.**

- повторно запускний валідатор:
  `scripts/validate-yefvv-2024.mjs`;
- формальна JSON Schema:
  `schemas/yefvv-it-question-dataset.schema.json`;
- машинний звіт:
  `reports/yefvv-it-2024-validation.json`.

Валідатор виконує 12 груп перевірок: синтаксис JSON, безпечний розбір YAML,
синтаксис схеми, еквівалентність YAML і JSON, відповідність прикладній
схемі, унікальність `id`, відсутність службового маркера `(x)`, наявність
усіх файлових посилань, ручну перевірку візуальних завдань, повне покриття
посторінковою ручною звіркою, checksum вихідного PDF та інвентаризацію
типів контент-блоків.

Поточний набір пройшов усі 12 перевірок:

- 140 завдань і 560 варіантів;
- 54 валідні файлові посилання;
- 17 із 17 візуально залежних завдань мають завершену ручну перевірку;
- 140 із 140 завдань звірено з усіма 29 рендерами сторінок і 54
  пов'язаними медіаресурсами;
- SHA-256 PDF збігається із зафіксованим значенням;
- використано лише блоки `markdown`, `math`, `code`, `table` та `image`;
- помилок валідації немає.

Скрипт завершується ненульовим кодом у разі будь-якої помилки та завжди
записує адресну діагностику в JSON-звіт, тому його можна використовувати
у CI.

### Ручна посторінкова валідація
