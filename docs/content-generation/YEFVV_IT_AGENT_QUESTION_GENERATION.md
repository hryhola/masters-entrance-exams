# Генерація додаткових питань ЄФВВ з інформаційних технологій

**Instruction ID:** `yefvv-it-agent-question-generation`  
**Version:** `1.0.0`  
**Target exam:** вступна кампанія 2026 року  
**Execution environment:** Codex, Claude Code або інший coding agent з
доступом до репозиторію та web-пошуку

## Як запустити

Надішли агенту короткий запит:

```text
Прочитай і виконай
docs/content-generation/YEFVV_IT_AGENT_QUESTION_GENERATION.md.

Параметри:
- topic_codes: [8.1.1]
- count: 10
- difficulty: medium
- delivery: draft

Не показуй у чаті питання, варіанти, ключі чи пояснення.
```

Допустимі параметри:

- `topic_codes`: один або кілька точних кодів із taxonomy;
- `count`: кількість питань, ціле число;
- `difficulty`: `easy`, `medium`, `hard` або `balanced`;
- `delivery`: `draft` або `publish`;
- `notes`: додаткові обмеження користувача.

Поки стандартне підключення generated dataset до content registry ще не
реалізоване, використовуй `delivery: draft`. Не імпровізуй production
integration.

## Роль агента

Ти створюєш нові тренувальні питання в академічному стилі предметного
тесту ЄФВВ з інформаційних технологій. Питання мають перевіряти знання та
застосування понять з офіційної програми, але не відтворювати офіційні
питання.

Ти відповідаєш одночасно за:

1. аналіз програми;
2. дослідження предметної основи;
3. проєктування питань;
4. правильність ключів;
5. пояснення;
6. перевірку стилю й унікальності;
7. створення dataset і звітів.

Не використовуй окремий model API, локальний LLM runtime або постійний
CLI-генератор. Працюй власними можливостями агента, web-пошуком і
звичайними інструментами репозиторію.

## Режим без спойлерів

Це обов'язкова вимога.

- Не виводь у чат формулювання нових питань.
- Не виводь варіанти відповідей.
- Не повідомляй правильні ключі.
- Не цитуй згенеровані пояснення.
- Не показуй diff або фрагменти generated dataset у фінальній відповіді.
- Записуй контент безпосередньо у файли репозиторію.
- У фінальній відповіді повідомляй лише batch ID, параметри, кількість,
  покриття тем, джерела, статус перевірок і шляхи до артефактів.

Якщо користувач прямо попросить показати конкретне generated питання,
можна виконати цей окремий запит. Інакше зберігай режим без спойлерів.

## Перевірка запиту

До початку генерації:

1. Переконайся, що кожний `topic_code` існує в taxonomy.
2. Зафіксуй назву розділу, назву теми й очікуваний cognitive level.
3. Дотримуйся `restrictions.prohibited_in_test_items`.
4. Не використовуй конкретну мову програмування, псевдокод або фрагмент
   програми, якщо програма це забороняє.
5. Врахуй виняток для SQL лише в темах, де програма прямо вимагає
   побудову SQL-запитів.
6. Якщо запит виходить за межі програми, зупинись без створення batch.

## Web-дослідження

Web-пошук є обов'язковим. Не покладайся лише на пам'ять моделі.

### Пріоритет джерел

1. Офіційна програма та матеріали `testportal.gov.ua`.
2. Стандарти, RFC, специфікації, офіційна технічна документація.
3. Офіційні документи державних органів для українського контексту.
4. Рецензовані наукові публікації та університетські підручники.

Не використовуй як предметну основу:

- SEO-статті;
- випадкові блоги;
- форуми та Q&A;
- неперевірені конспекти;
- інші AI-generated summaries;
- сайти з готовими тестами без простежуваних джерел.

Для кожної теми використовуй щонайменше два авторитетні джерела. Кожне
питання повинно мати хоча б одне джерело, яке прямо підтверджує правильну
відповідь. Для спірних або залежних від версії тверджень потрібні два
незалежні джерела.

Для фактів, які могли змінитися, перевір актуальність на день генерації.
Уникай version-sensitive деталей, якщо версія або дата не зазначена в
умові.

## Research report

Створи:

```text
reports/generated/<batch-id>.research.md
```

Звіт повинен містити:

- batch ID і параметри;
- точні topic codes;
- перелік джерел із URL;
- назву документа або сторінки;
- організацію чи автора;
- дату публікації або оновлення, якщо доступна;
- дату доступу;
- короткий перелік тверджень, які джерело підтверджує;
- відповідність `question_id -> source_id`;
- застереження щодо версій або термінології.

Не копіюй у research report повний текст generated питань.

## Проєктування batch

До написання формулювань створи внутрішню матрицю:

- topic code;
- cognitive level;
- difficulty;
- перевірюване поняття;
- тип хибного міркування для кожного distractor;
- source IDs;
- запланована позиція правильного варіанта.

Не показуй цю матрицю користувачеві.

Для `difficulty: balanced` використовуй приблизний розподіл:

- 30% easy;
- 50% medium;
- 20% hard.

За малої кількості питань округлюй так, щоб середній рівень залишався
домінантним.

Розподіляй правильні варіанти `a`, `b`, `c`, `d` максимально рівномірно.
Різниця між найчастішою і найрідшою позицією не повинна перевищувати 1,
якщо `count >= 4`.

## Вимоги до питання

Кожне питання повинно:

- мати рівно одну безумовно правильну відповідь;
- містити чотири взаємовиключні варіанти;
- відповідати одному з переданих topic codes;
- перевіряти змістовне поняття, а не дрібну trivia;
- бути самодостатнім без прихованого контексту;
- мати достатньо даних для розв'язання;
- використовувати сучасну українську академічну термінологію;
- бути граматично й стилістично узгодженим з усіма варіантами;
- уникати підказок через довжину, категорію або деталізацію відповіді.

Не використовуй:

- «усі відповіді правильні» або «жодна відповідь»;
- навмисні граматичні пастки;
- подвійне заперечення;
- слова «завжди» і «ніколи», якщо вони не є предметом перевірки;
- негативне формулювання на кшталт «що НЕ…», якщо позитивне можливе;
- гумор, розмовний стиль або культурно залежні натяки;
- конкретні назви продуктів, якщо тема перевіряє загальний принцип;
- твердження, правильність яких залежить від незафіксованої версії.

## Академічний стиль

Орієнтуйся на стиль українського вступного фахового тестування:

- нейтральне формулювання;
- термінологічна точність;
- коротка умова без зайвої історії;
- одна навчальна проблема на питання;
- distractors одного рівня абстракції;
- відсутність оцінних або рекламних формулювань.

Не імітуй конкретний університет і не стверджуй, що питання походить від
МОН, УЦОЯО або закладу вищої освіти.

## Distractors

Кожний неправильний варіант має бути правдоподібним для студента, який:

- плутає суміжні поняття;
- неправильно застосовує правильне правило;
- пропускає важливу передумову;
- використовує типову, але хибну узагальнену тезу.

Не створюй безглуздих варіантів лише для заповнення кількості.

## Ключ і пояснення

Для кожного питання:

1. Розв'яжи його незалежно від початкового задуму.
2. Підтвердь ключ джерелом.
3. Перевір, що жодний distractor не стає правильним за розумної
   інтерпретації.
4. Для обчислень повтори розрахунок іншим способом.
5. Для SQL перевір синтаксис і результат на мінімальному прикладі.
6. Для стандартів і протоколів перевір точне нормативне формулювання.

Пояснення повинно:

- прямо називати принцип, що робить ключ правильним;
- коротко пояснювати, чому distractors не відповідають умові;
- бути достатнім для навчання;
- не посилатися на «відповідь вище» або процес генерації;
- мати статус `generated`.

Якщо правильність не можна довести однозначно, перепиши або відкинь
питання.

## Захист від копіювання

Порівняй кожне нове питання з усіма питаннями
`data/yefvv-it-2024.json`.

Обов'язкові правила:

- не допускай дослівного збігу 10 послідовних нормалізованих слів, крім
  неминучих стандартних термінів;
- обчисли token-trigram Jaccard similarity для об'єднаного тексту умови
  та варіантів;
- зафіксуй максимальний score і найближчий official question ID;
- використовуй threshold `0.35`;
- виконай окрему semantic review: заміна кількох слів не робить питання
  новим, якщо перевіряється той самий факт у тій самій конструкції з
  аналогічними distractors.

Якщо lexical score перевищує threshold або semantic review виявляє
перефразовану копію, відкинь питання.

Не вигадуй similarity score. Збережи метод і фактичний результат у
validation report.

## Dataset

Batch ID:

```text
generated-yefvv-it-<topic-slug>-<YYYYMMDD>-<sequence>
```

Для `delivery: draft` створи:

```text
data/generated/drafts/<batch-id>.json
```

`delivery` визначає лише підключення до сайту. Навіть draft повинен
пройти всі checks і мати release status `ready_for_application`; він
залишається неопублікованим, доки його не додано до content registry.

Використовуй schema v2 з `Task`, `AssessmentItem` і `single_choice`.
Кожне питання є окремим task з одним assessment item.

Обов'язкові metadata:

```json
{
  "origin": "generated",
  "generation": {
    "batch_id": "<batch-id>",
    "agent": "codex | claude_code | other",
    "model": "<model, якщо відомо>",
    "instructions": {
      "id": "yefvv-it-agent-question-generation",
      "version": "1.0.0",
      "sha256": "<sha256 цього файла>"
    },
    "generated_at": "<ISO 8601>",
    "workflow_version": "1.0.0",
    "research_report": "reports/generated/<batch-id>.research.md",
    "parameters": {
      "topic": "<назва теми>",
      "difficulty": "easy | medium | hard",
      "task_type": "single_choice"
    }
  }
}
```

Якщо `difficulty: balanced`, сформуй окремий batch для кожного рівня або
використай значення dominant difficulty й зафіксуй фактичний розподіл у
звіті. Не додавай невідоме enum-значення до dataset.

Кожний assessment item повинен мати:

```json
{
  "answer": {
    "correct_choice": "a | b | c | d",
    "source": "generated_key"
  },
  "explanation": {
    "status": "generated",
    "summary": []
  },
  "source": {
    "generation_batch_id": "<batch-id>"
  }
}
```

`summary` фактично має бути непорожнім масивом content blocks.

## Validation report

Створи:

```text
reports/generated/<batch-id>.validation.json
```

Звіт повинен містити:

- batch ID;
- instruction ID, version і SHA-256;
- agent і model, якщо відомо;
- час перевірки;
- результат кожного check;
- question-level status без повного тексту питань;
- джерела для кожного ключа;
- розподіл правильних варіантів;
- lexical similarity method, threshold, maximum score і nearest ID;
- semantic duplicate review;
- знайдені проблеми та виконані виправлення;
- фінальний статус `passed` або `failed`.

Обов'язкові checks:

```text
schema
source_grounding
answer_integrity
explanation_integrity
duplicate_detection
official_similarity
exam_style
```

У release metadata використовуй:

```json
{
  "verification": {
    "method": "agent_validation",
    "status": "passed",
    "workflow_version": "1.0.0",
    "validated_at": "<ISO 8601>",
    "report": "reports/generated/<batch-id>.validation.json",
    "checks": [
      "schema",
      "source_grounding",
      "answer_integrity",
      "explanation_integrity",
      "duplicate_detection",
      "official_similarity",
      "exam_style"
    ],
    "similarity": {
      "maximum_score": 0,
      "threshold": 0.35
    }
  }
}
```

Не встановлюй `status: passed`, якщо хоча б один check не завершений або
не має доказів.

## Технічна перевірка

Після створення артефактів:

1. Запусти валідатор schema v2.
2. Перевір унікальність усіх ID.
3. Перевір наявність правильного choice у options.
4. Перевір рівно чотири choices для кожного питання.
5. Перевір непорожні пояснення.
6. Перевір збіг `generation_batch_id`.
7. Перевір баланс ключів.
8. Виконай similarity check.
9. Запусти релевантні тести проєкту.
10. Для `delivery: publish` також запусти повний `npm run check` і
    production build.

Не змінюй unrelated файли та не перезаписуй зміни користувача.

## Умови зупинки

Не публікуй batch, якщо:

- тема відсутня в офіційній програмі;
- web-пошук або авторитетні джерела недоступні;
- ключ хоча б одного питання неоднозначний;
- джерела суперечать одне одному;
- similarity threshold перевищено;
- питання порушує заборони програми;
- schema або тести не проходять;
- required report неповний.

Можна зберегти failed draft і validation report, але не підключати його до
сайту.
