# Етап 7. Згенерований контент і стабільний реліз

## Ціль

Додати контрольоване джерело нових завдань і підготувати сайт до
регулярного використання без змішування офіційного та згенерованого
контенту.

## Роботи з контентом

- визначити agent-driven workflow генерації без окремого API або локального
  model runtime;
- зберігати версіоновану інструкцію, яку безпосередньо виконує Codex,
  Claude Code або інший coding agent;
- генерувати за іспитом, темою, складністю й типом без показу нових питань
  користувачеві в чаті;
- перевіряти схему, відповідь і пояснення;
- досліджувати предметну основу через актуальні авторитетні джерела;
- перевіряти дублікати й схожість з офіційними питаннями;
- публікувати тільки набори, що пройшли всі обов'язкові agent checks;
- зберігати research report і validation report;
- маркувати `origin: generated`;
- зберігати назву агента, модель, версію інструкції та дату генерації;
- виключати близьке копіювання офіційних формулювань.

Окремий CLI-генератор, інтеграція з model API та локальний LLM runtime не
потрібні. Агент працює безпосередньо з файлами репозиторію, офіційною
програмою, наявними прикладами та web-пошуком.

Ручний перегляд питань користувачем не є частиною workflow. Це дозволяє
вперше побачити нове питання безпосередньо під час тесту. Агент не
виводить у чат формулювання, варіанти, ключі або пояснення, а записує їх у
dataset. У відповіді він повідомляє лише параметри batch, кількість,
використані джерела та результат перевірок.

Це не є офіційною або редакторською перевіркою. Кожен опублікований набір
містить машинозчитувані provenance, research report і validation report,
а інтерфейс прямо показує статус «Перевірено агентом».

## План реалізації

1. **Реалізовано.** Розширити модель даних походженням, agent provenance
   та рівнем перевірки.
2. **Реалізовано.** Додати версіоновану інструкцію для генерації через
   Codex або Claude Code без розкриття питань у чаті.
3. **Реалізовано.** Додати структуру generated dataset і підключення готового
   batch до content registry.
4. Закріпити в CI механічні quality gates: схема, ключ, пояснення,
   source grounding, дублікати, схожість і формат конкретного іспиту.
5. Додати видиме маркування, фільтри практики й окрему статистику за
   походженням.
6. Реалізувати резервне відновлення, міграції та політику оновлення
   dataset.
7. Завершити browser-, accessibility-, mobile- та release-перевірки.

## Agent checks

Generated batch не може мати статус `passed`, доки агент не виконав і не
задокументував такі перевірки:

- schema;
- source grounding;
- answer integrity;
- explanation integrity;
- duplicate detection;
- official similarity;
- exam style.

Поки відповідний механічний quality gate ще не реалізовано в CI, агент
мусить виконати перевірку самостійно, описати метод і докази у validation
report та не вигадувати числовий результат.

## Роботи зі стабілізації

- додати наскрізні browser-тести;
- перевірити accessibility;
- оптимізувати завантаження наборів і зображень;
- додати сторінку про джерела й обмеження;
- перевірити мобільні браузери;
- додати резервне відновлення прогресу;
- документувати release checklist;
- зафіксувати політику оновлення dataset і міграцій.

## Результат

Користувач може продовжувати практику після вивчення офіційних тестів,
чітко бачачи походження й рівень перевірки кожного нового питання.

## Критерії приймання

- згенерований набір без повних research і validation reports зі статусом
  `passed` не потрапляє в production;
- офіційні та згенеровані питання мають різне видиме маркування;
- для згенерованого питання видно, що його перевірив агент;
- статистику можна фільтрувати за походженням;
- production build проходить unit, integration і browser-тести;
- повторний GitHub Pages deploy не пошкоджує локальний прогрес;
- основні сценарії відповідають узгодженим вимогам доступності.
