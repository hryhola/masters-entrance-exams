# ЄВІ: аналіз джерел і модель даних

## Статус

Проведено первинний структурний аналіз усіх PDF ЄВІ з папки `raw`.
Схему v2 та контрольні fixtures реалізовано. Конвертацію повного набору
ще не розпочато.

## Реалізований прототип

Перший практичний крок завершено:

- додано модель `Stimulus`, `Task`, `AssessmentItem`;
- додано валідатор схеми v2 і перевірку посилань між сутностями;
- ідентифікатори відповідей у v2 є довільними рядками `ChoiceId`;
- реалізовано `matching_choice`, `cloze_choice` і `single_choice`;
- додано реальні fixtures: Task 1 англійської, мікротекст ТЗНК і
  ситуацію 25–27 з двома діаграмами;
- діаграми збережено як оригінальні зображення та продубльовано
  структурованими таблицями;
- додано інтерактивну сторінку `/dev/evi-fixtures`.

Fixture зберігається в
[`data/fixtures/evi-schema-v2.json`](../../../data/fixtures/evi-schema-v2.json).
Старий dataset ЄФВВ продовжує використовувати схему v1 без міграції.

Поточна межа: v2 ще не підключено до повних практичних сесій,
LocalStorage, історії та інтервального повторення. Це має виконуватися
після конвертації першого повного набору ТЗНК.

## Англійська мова 2023

Усього 30 оцінюваних відповідей, кожна дає 0 або 1 бал.

| PDF-сторінка | Структура                                                 | Оцінювані елементи |
| -----------: | --------------------------------------------------------- | -----------------: |
|            1 | титульна сторінка і склад тесту                           |                  — |
|            2 | Task 1: шість текстів і вісім спільних тверджень A–H      |                1–6 |
|            3 | Task 2: один текст і п'ять питань з вибором A–D           |               7–11 |
|            4 | Task 3: один текст із десятьма пропусками та окремими A–D |              12–21 |
|            5 | Task 4: два тексти з дев'ятьма пропусками та окремими A–D |              22–30 |
|            6 | офіційний ключ                                            |                  — |

Підтверджені типи взаємодії:

1. `matching`: для кожного тексту вибирається одне зі спільних тверджень
   A–H; два твердження зайві, повторне використання відповіді заборонене.
2. `single_choice`: звичайне питання A–D до спільного тексту.
3. `cloze_choice`: вибір A–D для кожного пропуску всередині спільного
   тексту.

Task 1 не можна коректно моделювати шістьма незалежними
`single_choice`, оскільки всі підзавдання використовують спільний набір
варіантів і обмеження унікальності.

## Проблема поточної моделі

Поточний dataset прирівнює один екран, одне питання й одну оцінювану
відповідь:

```text
Question -> prompt + options + correctOption
```

Для ЄВІ це не завжди так:

- один великий текст використовується кількома питаннями;
- один task може містити кілька оцінюваних пропусків;
- кілька підзавдань можуть використовувати один набір відповідей;
- діаграма належить ситуації, а не окремому питанню.

## Запропонована модель

```text
Dataset
  Section
  Stimulus
  Task
    AssessmentItem
```

### `Stimulus`

Повторно використовуваний контекст:

- текст або пара текстів;
- мікротекст із позначеними пропусками;
- логічна ситуація та її умови;
- діаграма;
- таблиця або математичні дані.

`Stimulus.content` використовує наявні `ContentBlock`. Для складної
діаграми зберігаємо:

1. оригінальний crop як `image`;
2. точний `alt`;
3. структуровану таблицю даних, якщо її можна надійно відновити.

### `Task`

Одиниця показу та навігації. Містить інструкцію, посилання на стимули й
один або кілька оцінюваних елементів.

### `AssessmentItem`

Одиниця відповіді, оцінювання, історії та інтервального повторення.
Саме її ідентифікатор має використовувати `QuestionProgress`.

Потрібні типи відповіді:

```text
single_choice
matching_choice
cloze_choice
```

Множинна правильна відповідь, короткий текст і довільне заповнення
пропуску в наявних PDF не підтверджені, тому на цьому етапі їх не
реалізовуємо.

## Приклад YAML: matching

```yaml
tasks:
  - id: yevi-en-2023-task-01
    number: 1
    type: matching
    instruction:
      - type: markdown
        text: Match choices A–H to texts 1–6.
    choice_set:
      id: library-features
      unique: true
      choices:
        - id: a
          content:
            - type: markdown
              text: contains the founder's book collection
        - id: b
          content:
            - type: markdown
              text: is named after its designer
    items:
      - id: yevi-en-2023-001
        number: 1
        response:
          type: matching_choice
          choice_set_id: library-features
        stimulus_ids: [library-biblioburro]
        answer:
          correct_choice: f
```

## Приклад YAML: група пропусків

```yaml
stimuli:
  - id: tznk-2024-microtext-02
    type: cloze_text
    content:
      - type: markdown
        text: >
          Гендер є флюїдним, {{blank:2}}, адже такою є поведінка...
          Визначеність тягне за собою {{blank:3}}.

tasks:
  - id: tznk-2024-task-02
    number: 2
    type: cloze
    stimulus_ids: [tznk-2024-microtext-02]
    items:
      - id: tznk-2024-002
        display_label: "2.(2)"
        response:
          type: cloze_choice
          blank_id: "2"
          options: [a, b, c, d]
        answer:
          correct_option: b
      - id: tznk-2024-003
        display_label: "2.(3)"
        response:
          type: cloze_choice
          blank_id: "3"
          options: [a, b, c, d]
        answer:
          correct_option: c
```

## Зміни в застосунку

1. Розширити `ExamDefinition`:
   - секції і порядок проходження;
   - кількість оцінюваних елементів;
   - правила навігації між tasks;
   - окремі правила таймера й підрахунку балів.
2. Замінити `OptionId = a | b | c | d` на рядковий ідентифікатор,
   валідований конкретним набором відповідей.
3. Замінити `answers: Record<string, OptionId>` на
   `Record<assessmentItemId, AnswerValue>`.
4. Навігацію сесії вести за `Task`, а прогрес і score рахувати за
   `AssessmentItem`.
5. Додати renderers:
   - `SingleChoiceTask`;
   - `MatchingTask`;
   - `ClozeTask`.
6. Показувати спільний stimulus один раз, а не дублювати в кожному
   питанні.
7. Додати `lang="en"` до англомовних текстів і відповідну типографіку.
8. Мігрувати локальний прогрес, залишивши ключі ЄФВВ сумісними.

## Рекомендація

Не конвертувати всі три PDF одним запитом. Безпечний процес:

1. детерміноване витягування;
2. структурний draft по одному task або групі;
3. перевірка ключа;
4. перевірка сторінки PDF;
5. лише потім об'єднання в dataset.

Схема v2 і контрольні fixtures підтвердили модель та UI. Наступний
практичний крок: розпочати етап C з конвертації пропусків 1–10 ТЗНК
2024, після чого провести звірку з ключем і сторінками PDF.
